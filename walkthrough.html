CS 4880 P4 — Compiler Walkthrough

CMP_SCI 4880 · Compiler Design · Code Generation Project

Introduction

This project implements a complete compiler for a small imperative language called EXT (source files use the .ext extension). The compiler translates EXT source code into assembly for a simple accumulator-based virtual machine (VM). The pipeline was built incrementally across four projects: P1 (lexer), P2 (parser), P3 (static semantics), and P4 (code generation).

The codebase has six C source files plus headers, each responsible for exactly one compiler phase. The entry point is P4.c; it wires all phases together and drives the overall process.

Project Spec → docs/Projects View_ P4_*.pdf The assignment spec defines what the EXT language looks like, what the expected output format is, and grading criteria. Every grammar rule, token constraint, and error message format in this code traces directly to that spec.

Each phase feeds the next. The lexer produces a token stream; the parser consumes that stream and builds an Abstract Syntax Tree (AST); the semantic pass walks the AST to enforce scope/type rules; code generation walks the same AST and emits VM assembly. If any phase detects an error it prints a message and exits immediately — there is no error recovery.

Target VM

The generated .asm files run on a register-based virtual machine that exposes one accumulator register (ACC) and a flat memory of named variables. Every arithmetic instruction operates on ACC and one memory location; there are no general-purpose registers.

VM definition → docs/VM_Language.pdf Defines every instruction, its operand format, and execution semantics. All VM instructions used in codegen.c — LOAD, STORE, ADD, SUB, MULT, DIV, READ, WRITE, STOP, BR, BRNEG, BRPOS, BRZERO, BRZPOS, BRZNEG, NOOP — come directly from this document.

token.h — Token Types

Every piece of the compiler that needs to talk about tokens includes this header. It defines exactly three things: the tokenID enum, the Token struct, and the max instance length constant. 

typedef enum {
    IDTk,      // identifier: starts with 'x', ≤8 chars
    IntTk,     // integer literal: decimal digits, ≤8 chars
    KW_tk,     // keyword: go og loop int exit scan output cond set...
    OpTk,      // multi-char operator: ?le ?ge ?lt ?gt ?ne ?eq ** //
    OpDelTk,   // single-char operator/delimiter: + - : ; = ( ) { } [ ]
    EOFTk
} tokenID;

#define TOKEN_INSTANCE_MAX 9   // 8 significant chars + NUL

typedef struct {
    tokenID  id;
    char     instance[TOKEN_INSTANCE_MAX];
    int      lineNumber;
} Token;

The instance field stores the actual text of the token (e.g., "xcount" for an identifier, "42" for an integer). The lineNumber field is used to produce accurate error messages in all phases.

Language spec → docs/Projects View_ P4_*.pdf The six token types map one-to-one to the categories defined in the spec. The 8-character limit on identifiers and integers is a language-level constraint from the spec, enforced here via TOKEN_INSTANCE_MAX = 9 (8 chars + null terminator).

lexer.c — Tokenizer

The lexer reads characters from a FILE*, classifies them into tokens, and returns one token per call to getNextToken(). It maintains a single line counter (lineNum) shared across all calls.

Initialization

static FILE *src     = NULL;
static int   lineNum = 1;

void lexer_init(FILE *fp) {
    src     = fp;
    lineNum = 1;
}

lexer_init must be called once before any tokens are requested. The file pointer and line number are kept as module-level statics so getNextToken doesn't need parameters on every call.

Character-Level Helpers

static int nextChar(void) {
    int c = fgetc(src);
    if (c == '\n') lineNum++;
    return c;
}

static void pushBack(int c) {
    if (c == '\n') lineNum--;
    ungetc(c, src);
}

All character reads go through nextChar so the line counter stays accurate. pushBack mirrors ungetc but decrements the counter when a newline is un-read, preventing double-counting. The one-character look-ahead is enough because every token type is distinguishable from its first character (or first two for ** and //).

Comment Handling

if (c == '@') {
    int c2;
    while (1) {
        c2 = nextChar();
        if (c2 == EOF)        lexError("unterminated comment");
        if (isspace(c2))      lexError("whitespace inside comment");
        if (c2 == '@')        break;
    }
    goto restart;   // O(1) — avoids stack growth for consecutive comments
}

EXT comments are delimited by @...@ with no internal whitespace allowed (e.g., @startofprogram@). After consuming the closing @, the code uses goto restart rather than calling getNextToken recursively. A recursive call would grow the call stack by one frame per comment; the goto resets the local frame in O(1) space.

Identifier and Keyword Recognition

if (isalpha(c)) {
    // ... accumulate up to 8 chars ...
    if (isKeyword(buf))
        return makeToken(KW_tk, buf, tokenLine);
    if (buf[0] != 'x')
        lexError("invalid identifier");     // must start with 'x'
    return makeToken(IDTk, buf, tokenLine);
}

The lexer first builds the full word, then checks if it is a keyword. This matters because all keywords start with letters other than x (go, int, loop, etc.) so a non-x start that isn't a keyword is a lexical error. Identifiers must start with x — this is an unusual language constraint that prevents variable names from shadowing keywords.

Multi-Character Operators

if (c == '?') {
    // consumes exactly 2 more chars: e.g., ?ge ?lt
    buf[0]='?'; buf[1]=c2; buf[2]=c3; buf[3]='\0';
    if (valid) return makeToken(OpTk, buf, tokenLine);
}

if (c == '*') {
    c2 = nextChar();
    if (c2 == '*') return makeToken(OpTk, "**", tokenLine);
    pushBack(c2);
    lexError("bare '*' is not a valid token");
}

?-prefixed relational operators are always exactly 3 characters. ** (multiply) and // (divide) are exactly 2. A bare * or / is a lexical error — the language has no single-character multiply/divide.

tree.h / tree.c — AST Nodes

The AST is built from a single Node struct. Every grammar non-terminal becomes one node, identified by a string label. 

#define MAX_CHILDREN 4

typedef struct Node {
    char  label[32];           // non-terminal name: "program", "block", "exp", ...
    Token tokens[3];           // terminal tokens attached to this node
    int   tokenCount;
    struct Node *children[MAX_CHILDREN];
    int   childCount;
} Node;

A node stores up to 3 tokens (enough for any grammar rule — the most tokens a single rule attaches is 2 for vars: identifier + integer). It stores up to 4 children (enough for program → vars block and similar rules). The exact capacity choices come from manually counting the maximum tokens/children any single grammar rule can attach.

Key Functions

FunctionPurpose
newNode(label)Allocates and zero-initializes a node
addChild(parent, child)Appends a child; silently ignores NULL children (for empty productions)
addToken(n, t)Attaches a terminal token to a node
freeTree(n)Post-order recursive free — frees children before the parent node
Note: addChild silently skips NULL children. This is intentional — grammar rules with empty productions (like <vars> → empty) return NULL from the parser, and the caller passes that NULL to addChild without needing an if guard at every call site.

parser.c — Recursive Descent Parser

The parser implements the EXT grammar using a classic recursive descent strategy: one C function per non-terminal, each returning a Node*. It uses a single token of lookahead (static Token tk) and the "delay technique" for right-recursive rules.

Global Lookahead Token

static Token tk;   // always holds the next unprocessed token

static void consume(void) { tk = getNextToken(); }

Every function reads tk to decide which production to apply. After using a terminal it calls consume() to advance. The parser's public entry point, parser(), loads the first token before calling program_rule().

Helper Predicates

static int isKW   (const char *kw)  { return tk.id == KW_tk    && strcmp(tk.instance, kw) == 0; }
static int isOpDel(const char *op)  { return tk.id == OpDelTk  && strcmp(tk.instance, op) == 0; }
static int isOp   (const char *op)  { return tk.id == OpTk     && strcmp(tk.instance, op) == 0; }

These small predicates make the grammar rules readable. Without them, every check would be a two-part conjunction of tk.id == ... and strcmp(...).

The Delay Technique for Right-Recursive Rules

// <exp> -> <M> ** <exp> | <M> // <exp> | <M>
static Node *expr(void) {
    Node *n = newNode("exp");
    addChild(n, M());                    // always parse <M> first
    if (isOp("**") || isOp("//")) {   // then "delay" decision: is there an operator?
        addToken(n, tk);
        consume();
        addChild(n, expr());
    }
    return n;
}

The grammar for expressions is right-recursive. A pure LL(1) approach would require left-factoring the grammar into expr' → op M expr' | ε. Instead, this uses the "delay technique": parse the mandatory left part first, then peek at the lookahead to decide whether the optional right part is present. This produces the same parse tree without rewriting the grammar, and it is the approach recommended by the course docs.

P4 Suggestions → docs/P4 Suggestions_*.pdf The suggestion document recommends the delay technique for all right-recursive expression rules (exp, M, N). Code generation relies on the resulting AST shape — specifically on n->tokenCount and the number of children — to decide which code pattern to emit.

The = = Relational Operator

static Node *relational(void) {
    Node *n = newNode("relational");
    if (tk.id == OpTk) {
        addToken(n, tk); consume();        // ?le ?ge etc.
    } else if (isOpDel("=")) {
        addToken(n, tk); consume();        // first '='
        if (!isOpDel("=")) error("second '='");
        addToken(n, tk); consume();        // second '='
    } ...
}

Equality is written as = = (two tokens with whitespace). The lexer produces two OpDelTk tokens each containing "=". Both are stored on the relational node. Code gen detects this case by checking n->tokenCount == 2, which signals that two branch instructions are needed (see Relational Branches).

semantics.c — Static Semantic Analysis

The semantic pass does a single preorder walk of the AST enforcing three rules from the language spec:

  1. Every identifier used must have been declared — error.
  2. No identifier may be declared more than once — error.
  3. Every declared identifier must be used at least once — warning.

Symbol Table

#define STV_MAX 100

typedef struct {
    char name[9];   // 8 chars + NUL, matching TOKEN_INSTANCE_MAX
    int  used;
} STVEntry;

static STVEntry stv[STV_MAX];
static int      stv_size = 0;

The symbol table is a flat array — no hash table, no scoping. EXT has no nested scope semantics beyond the fact that declarations in a block are accessible throughout the block. The project spec explicitly allows a flat table with a maximum of 100 entries.

Scope limitation: Because the symbol table is flat and never shrinks, an identifier declared in an inner block technically stays in the table for the rest of the program. The spec says this is acceptable for P3/P4; a production compiler would use a scope stack.

The Walk

static void walk(Node *n) {
    if (!n) return;

    if (isDefNode(n->label)) {     // "vars" or "varList" → declaration
        for (i ...) if (IDTk) insert(name);
    } else {                        // any other node → use site
        for (i ...) if (IDTk) verify(name);
    }

    for (i ...) walk(n->children[i]);  // recurse into children
}

The key insight: only vars and varList nodes carry declaration tokens. Every other node that holds an IDTk token is a use site. The walk processes all tokens on a node before recursing, which means a node's own tokens are checked before its subtree — effectively a preorder traversal for declarations, which ensures the spec's "declared before use" rule works because all declarations appear before the statement list in the grammar.

codegen.c — Code Generation (P4 Core)

This is the primary new module for P4. It takes the semantically-validated AST and emits VM assembly. The design follows the strategy in the P4 docs: one function per AST node type, each leaving its result in ACC, using temporary variables for sub-expression intermediates, and collecting all variable declarations at the end of the output as a flat data section.

Code Generation Support → docs/P4 Code Generation Support_*.pdf This document is the primary reference for codegen.c. It defines the mapping from each grammar construct to a sequence of VM instructions, the role of temporaries, and the two-phase structure (instructions then data). Every function in codegen.c corresponds to a section in this document.

Module State

static FILE *outFile;
static int   tempCounter;
static int   labelCounter;

typedef struct { char name[16]; int initVal; } VarEntry;
static VarEntry varTable[200];
static int      varCount;

All state is module-level and reset at the start of codeGen(). This makes the function safe to call multiple times (though the compiler only calls it once). The varTable serves double duty: it stores both user-declared variables (name + initial value) and compiler-generated temporaries (name + initial 0). Both end up emitted to the data section after STOP.

Storage Allocation → docs/P4 Storage Allocation Support_*.pdf Defines the two-region output layout: (1) instruction region at the top, (2) data region after STOP. All variables — user and compiler-generated — live in the data region as name initial_value lines. This is exactly what varTable accumulates.

Compiler Temporaries

static const char *newTemp(void) {
    snprintf(varTable[varCount].name, NAME_MAX, "t%d", tempCounter++);
    varTable[varCount].initVal = 0;
    return varTable[varCount++].name;
}

newTemp() allocates a fresh temporary — t0, t1, t2, … — by writing it directly into varTable and returning a pointer to the name within that table entry. The pointer is valid for the lifetime of the compiler run because varTable is a static array that never moves.

Design tradeoff: Temporaries are never reused. Every subexpression gets its own slot. This wastes memory slots but avoids the complexity of a liveness analysis / register allocator. The P4 docs explicitly allow this approach.

There is a subtle pattern used throughout the expression generators because the VM's arithmetic instructions require the operand to already be in memory, not in ACC: 

// To compute A + B:
LOAD  A          // ACC = A
STORE t0         // t0 = A  (spill so B can use ACC)
LOAD  B          // ACC = B
ADD   t0         // ACC = B + t0 = B + A ✓

Labels

static const char *newLabel(void) {
    static char buf[NAME_MAX];
    snprintf(buf, NAME_MAX, "L%d", labelCounter++);
    return buf;
}

Labels are named L0, L1, … and are used exclusively for branch targets in cond and loop constructs. Unlike newTemp, newLabel reuses a single static buffer — this is safe because every caller copies the returned string into a local array before calling newLabel again.

Variable Collection

static void genVars(Node *n) {
    if (!n) return;
    if (strcmp(n->label, "vars") == 0) {
        if (n->tokenCount >= 2)
            registerVar(n->tokens[0].instance, atoi(n->tokens[1].instance));
        if (n->childCount > 0) genVars(n->children[0]);
    } else if (strcmp(n->label, "varList") == 0) {
        if (n->tokenCount >= 2)
            registerVar(n->tokens[0].instance, atoi(n->tokens[1].instance));
        if (n->childCount > 0) genVars(n->children[0]);
    }
}

genVars traverses a vars/varList subtree and calls registerVar for each declared identifier, capturing the initial integer value from the AST. It does not emit any instructions — declarations produce no code, only data section entries. This matches the VM model where storage is pre-allocated, not pushed at runtime.

Storage Allocation → docs/P4 Storage Allocation Support_*.pdf Variable declarations in EXT always carry an initializer (e.g., int x = 5). The VM's data section format name init_value maps directly to this: the identifier and its integer initializer are both stored in the AST tokens and retrieved here by tokens[0] (name) and tokens[1] (value).

Expression Generation

Expressions are generated bottom-up. Each of the four expression functions (genR, genN, genM, genExp) leaves its result in ACC. The calling function can then immediately STORE it to a temporary if it needs the value later.

genR — Atoms

static void genR(Node *n) {
    if (n->childCount == 1)
        genExp(n->children[0]);          // parenthesized: ( exp )
    else
        emit("LOAD %s", n->tokens[0].instance);  // identifier or integer
}

An R node is either a parenthesized expression (recurse) or a leaf — an identifier or integer literal. The VM's LOAD instruction accepts both variable names and numeric literals, so both cases produce the same instruction.

genN — Unary and Binary Minus

static void genN(Node *n) {
    if (n->tokenCount == 0) { genR(n->children[0]); return; }

    if (strcmp(n->children[0]->label, "N") == 0) {
        // Unary minus: - <N>
        genN(n->children[0]);           // ACC = N
        strcpy(t, newTemp());
        emit("STORE %s", t);             // t = N
        emit("LOAD 0");                  // ACC = 0
        emit("SUB %s", t);               // ACC = 0 - N = -N
    } else {
        // Binary minus: <R> - <N> needs ACC = R - N
        genN(n->children[1]);           // ACC = N (RHS first)
        strcpy(t, newTemp());
        emit("STORE %s", t);             // t = N
        genR(n->children[0]);           // ACC = R
        emit("SUB %s", t);               // ACC = R - t = R - N ✓
    }
}

Subtraction is not commutative, so order matters. The pattern evaluate RHS first, spill to temp, evaluate LHS into ACC, then SUB the temp produces the correct ACC = LHS - RHS result. This same pattern repeats for division in genExp.

genM — Addition

static void genM(Node *n) {
    if (n->childCount == 1) { genN(n->children[0]); return; }
    // N + M — addition is commutative, evaluate N first
    genN(n->children[0]);
    strcpy(t, newTemp()); emit("STORE %s", t);
    genM(n->children[1]);
    emit("ADD %s", t);
}

Addition is commutative so the order of evaluation doesn't affect correctness, but the code still uses the spill-then-compute pattern for consistency. Either operand could be evaluated first — N is chosen arbitrarily.

genExp — Multiplication and Division

static void genExp(Node *n) {
    if (n->childCount == 1) { genM(n->children[0]); return; }

    if (strcmp(op, "**") == 0) {
        genM(n->children[0]);               // commutative: either order
        strcpy(t, newTemp()); emit("STORE %s", t);
        genExp(n->children[1]);
        emit("MULT %s", t);
    } else {
        // Division: ACC = left // right — NOT commutative
        genExp(n->children[1]);             // RHS first
        strcpy(t, newTemp()); emit("STORE %s", t);
        genM(n->children[0]);               // LHS into ACC
        emit("DIV %s", t);                  // ACC = LHS / RHS ✓
    }
}

Relational Branches

static const char *relBranchInstr(Node *rel, int *needTwo) {
    *needTwo = 0;
    const char *op = rel->tokens[0].instance;

    if (rel->tokenCount == 2)         // "= =" — two tokens
        { *needTwo = 1; return "BRNEG"; }

    if (strcmp(op, "?ge") == 0) return "BRNEG";   // LHS - RHS < 0 → NOT ?ge
    if (strcmp(op, "?le") == 0) return "BRPOS";   // LHS - RHS > 0 → NOT ?le
    if (strcmp(op, "?gt") == 0) return "BRZNEG";  // LHS - RHS <= 0 → NOT ?gt
    if (strcmp(op, "?lt") == 0) return "BRZPOS";  // LHS - RHS >= 0 → NOT ?lt
    if (strcmp(op, "?ne") == 0) return "BRZERO";  // LHS - RHS == 0 → NOT ?ne
    if (strcmp(op, ";")   == 0) return "BRZERO";  // ';' = ?ne alias
    if (strcmp(op, "?eq") == 0) { *needTwo = 1; return "BRNEG"; }
}

The key insight: the VM doesn't have "jump if LHS ≥ RHS" — it only has "jump if ACC is negative/positive/zero". The code generation strategy computes ACC = LHS - RHS before the branch, then branches on the negation of the condition to skip the body:

Source conditionBranch (false → skip)Condition of ACC
?ge (LHS ≥ RHS)BRNEGACC < 0 → LHS < RHS → false
?le (LHS ≤ RHS)BRPOSACC > 0 → LHS > RHS → false
?gt (LHS > RHS)BRZNEGACC ≤ 0 → LHS ≤ RHS → false
?lt (LHS < RHS)BRZPOSACC ≥ 0 → LHS ≥ RHS → false
?ne or ;BRZEROACC = 0 → LHS = RHS → false
?eq or = =BRNEG + BRPOSACC ≠ 0 → LHS ≠ RHS → false

Equality is the special case: the VM has no "branch if non-zero", so equality requires two branch instructions — one for negative and one for positive — both jumping to the exit label. needTwo signals this to the caller.

Runtime Semantics → docs/P4 Runtime Semantics_*.pdf This document defines what "the condition holds" means for each relational operator in terms of the ACC value after LHS - RHS. The branch-negation strategy implemented here is the direct consequence: generate code that branches out of the body when the condition is false.

Statement Dispatch

static void genNode(Node *n) {
    if (strcmp(n->label, "program") == 0)   { ... emit("STOP"); ... }
    else if (strcmp(n->label, "block") == 0)    { ... }
    else if (strcmp(n->label, "read") == 0)     { emit("READ %s", ...); }
    else if (strcmp(n->label, "print") == 0)    { genExp(...); STORE t; WRITE t; }
    else if (strcmp(n->label, "assign") == 0)   { genExp(...); STORE x; }
    else if (strcmp(n->label, "cond") == 0)     { genCondOrLoop(n, 0); }
    else if (strcmp(n->label, "loop") == 0)     { genCondOrLoop(n, 1); }
    ...
}

The print statement requires a temporary even though it only needs to write a value. This is because the VM's WRITE instruction takes a memory address (variable name), not an ACC value. The expression is evaluated into ACC, spilled to a temp, then that temp is WRITEn.

Cond and Loop — genCondOrLoop

static void genCondOrLoop(Node *n, int isLoop) {
    if (isLoop) { loopTop = newLabel(); emit("%s: NOOP", loopTop); }
    exitLabel = newLabel();

    genExp(n->children[1]);                    // ACC = RHS of condition
    strcpy(t, newTemp()); emit("STORE %s", t);
    emit("LOAD %s", n->tokens[0].instance);    // ACC = LHS (identifier)
    emit("SUB %s", t);                          // ACC = LHS - RHS

    brInstr = relBranchInstr(n->children[0], &needTwo);
    emit("%s %s", brInstr, exitLabel);
    if (needTwo) emit("BRPOS %s", exitLabel);

    genNode(n->children[2]);                   // body

    if (isLoop) emit("BR %s", loopTop);        // loop: jump back
    emit("%s: NOOP", exitLabel);
}

cond and loop share almost identical code structure. The only difference is that loop emits a label before the condition check and an unconditional BR back to it after the body, forming the back-edge of the loop. NOOP is used at all label targets because the VM requires every label to be attached to an instruction, and NOOP is a safe no-operation placeholder.

Storage Layout

// In genNode for "program":
emit("STOP");
for (int i = 0; i < varCount; i++)
    emit("%s %d", varTable[i].name, varTable[i].initVal);

After the final STOP instruction, all variables are emitted in declaration order. User variables come first (in the order genVars encountered them), then temporaries (in the order they were created). All temporaries are initialized to 0.

Storage Allocation → docs/P4 Storage Allocation Support_*.pdf The "data section after STOP" layout is mandated by the VM spec. The VM loader reads the instructions from the top of the file and the data declarations from after STOP. Temporaries must be declared here too — they are real memory slots, not CPU registers.

P4.c — Driver

int main(int argc, char *argv[]) {
    // 1. Parse args: optional -l flag, optional basename
    // 2. Open basename.ext for input (or stdin)
    lexer_init(in);

    if (lexOnly) {
        runLexer();                     // lex-only mode: print tokens
    } else {
        Node *tree = parser();          // parse → AST
        staticSemantics(tree);          // semantic check (errors exit here)

        // 3. Open basename.asm for output (or stdout)
        codeGen(tree, out);             // emit VM assembly
        fclose(out);
        freeTree(tree);
    }
}

The driver handles file I/O at both ends: it opens basename.ext for source input and basename.asm for assembly output. If no basename is given, stdin/stdout are used instead (handy for piping in tests). The -l flag runs the lexer only and dumps tokens — useful for debugging the tokenizer independently.

staticSemantics runs before code generation. If there are semantic errors, the process exits before any assembly is written — no partial output files are created for the error case. The tree is always freed before exiting.

Example: Print a Literal

Source (test_good1.ext)
go
{ output 1 :
}
exit
Output (test_good1.asm)

The output statement calls genExp → genM → genN → genR which emits LOAD 1 (the integer literal goes directly into ACC). The print handler then spills ACC to a temp (STORE t0) and WRITEs it. t0 is declared in the data section with initial value 0.

STORE t4
LOAD  0
SUB   t4       ← 0 - x = -x
STORE t5
WRITE t5
STOP

For x // 3 (division), the RHS (3) is evaluated first into ACC, spilled to t2, then the LHS (x) is loaded, and DIV t2 produces ACC = x / 3 — demonstrating the RHS-first pattern for non-commutative operations. Unary minus is implemented as 0 - x using the LOAD 0 / SUB trick.

Example: Conditional

Source (test_good8.ext)
go int x = 1 x1 = 2 :
{ scan x :
  cond [ x ?ge 0 ]
  output x :
}
exit
Output (test_good8.asm)
READ  x
LOAD  0
STORE t0      ← RHS = 0
LOAD  x       ← LHS
SUB   t0      ← ACC = x - 0
BRNEG L0     ← skip if x < 0 (condition ?ge fails)
LOAD  x
STORE t1
WRITE t1
L0: NOOP
STOP

The cond condition is x ?ge 0. After computing ACC = x - 0, the branch is BRNEG L0 — meaning "if x−0 is negative, skip the body." If x is ≥ 0, the branch is not taken, the body executes, then falls through to L0: NOOP.

Example: Loop

Source (test_good11.ext, loop portion)
loop [ x ; 0 ]
{ set x = x - 1 :
}
Generated assembly
L1: NOOP        ← loop top
LOAD  0
STORE t2
LOAD  x
SUB   t2       ← x - 0
BRZERO L2    ← exit when x = 0 (condition ; = ?ne fails)
LOAD  1
STORE t3
LOAD  x
SUB   t3
STORE x       ← x = x - 1
BR    L1      ← back-edge
L2: NOOP        ← loop exit

The ; relational operator is an alias for ?ne (not equal). The loop continues while x ≠ 0, so it exits when x = 0, which is when ACC = x - 0 = 0, triggering BRZERO. The back-edge BR L1 is emitted only for loop, not cond.

Example: Equality (= =) — Two Branches

Source
cond [ x = = 0 ]
output x :
Generated assembly
LOAD  0
STORE t0
LOAD  x
SUB   t0
BRNEG L0    ← if x < 0, not equal → skip
BRPOS L0    ← if x > 0, not equal → skip
... body (executes only when x = 0) ...
L0: NOOP

The VM has no single "branch if non-zero" instruction. Equality is synthesized with two branches: one for negative (not equal) and one for positive (not equal). Only when ACC is exactly zero (LHS = RHS) do both branches fail, letting execution fall through to the body.

Grammar Reference

<program>    -> go <vars> <block> exit
<vars>       -> empty | int identifier = integer <varList> :
<varList>    -> identifier = integer <varList> | empty
<block>      -> { <vars> <stats> }
<stats>      -> <stat> <mStat>
<mStat>      -> empty | <stat> <mStat>
<stat>       -> <read> | <print> | <block> | <cond> | <loop> | <assign>
<read>       -> scan identifier :
<print>      -> output <exp> :
<cond>       -> cond [ identifier <relational> <exp> ] <stat>
<loop>       -> loop [ identifier <relational> <exp> ] <stat>
<assign>     -> set identifier = <exp> :
<relational> -> ?le | ?ge | ?lt | ?gt | ?ne | ?eq | = = | ;
<exp>        -> <M> ** <exp> | <M> // <exp> | <M>
<M>          -> <N> + <M> | <N>
<N>          -> <R> - <N> | - <N> | <R>
<R>          -> ( <exp> ) | identifier | integer
